# Архитектура Rozitta Parser

Документ описывает, как пользователь выполняет ключевые задачи, какие модули и данные задействованы, и где сейчас возникают проблемы.

---
## 🗺️ [Интерактивная карта проекта (с заделом на две будущих версии интерфейса)](https://nynchezyabka.github.io/RozittaParser/map.html)

## Основные сценарии использования

### 1. Авторизация и подготовка

**Цель:** Пользователь входит в Telegram, чтобы программа получила доступ к его чатам.

**Поток:**
1. **UI (AuthScreen)** – пользователь вводит `api_id`, `api_hash`, номер телефона (или указывает путь к `tdata`).
2. **Воркер (AuthWorker)** запускает модуль `auth` (`features/auth/api.py`).
3. **Модуль `auth`** – подключается к Telegram API через Telethon, создаёт или загружает `.session` файл.
4. При успехе возвращает объект `User` с данными аккаунта.
5. **UI (MainWindow)** переключает состояние на «авторизован», разблокирует вкладку «Чаты».

**Задействованные модули:** `AuthScreen`, `AuthWorker`, `auth` (API).

**Передаваемые данные:** `AppConfig` → `User` (id, first_name, phone).

**Известные ограничения:**
- Импорт из `tdata` может не работать при включённом прокси (требуется временное отключение).  
  _Проблема: в `AuthService.import_from_tdata` не учитываются настройки прокси пользователя._

**Статус:** ✅ работает (кроме известного нюанса).

---

### 2. Выбор чата и загрузка списка

**Цель:** Пользователь видит список своих чатов, может выбрать нужный чат (или топик форума).

**Поток:**
1. **UI (ChatsScreen)** – инициирует загрузку через `ChatsWorker`.
2. **Модуль `chats` (`features/chats/api.py`)** – через Telethon получает список диалогов, нормализует ID, определяет тип чата (канал, форум, группа), сохраняет в `cached_dialogs` (SQLite).
3. Возвращает список словарей `ChatDict` (id, title, type, linked_chat_id, ...).
4. **UI (ChatsScreen)** показывает список; при клике на чат вызывается `chat_selected`.

**Дополнительно:** при выборе форума `TopicsWorker` загружает список топиков через модуль `chats`.

**Передаваемые данные:** `TelegramClient` → `List[ChatDict]`.

**Известные проблемы / баги:**
- Экспорт списка участников (топ активных) не учитывает комментаторов в каналах – показывает только из чата.  
  _Нужно: для каналов собирать авторов из linked группы комментариев._

**Статус:** ✅ основная функция работает. Баг с участниками – в плане исправления.

---

### 3. Парсинг сообщений (получение архива)

**Цель:** Пользователь выбирает период, типы медиа, фильтры (по автору, по теме) и запускает скачивание сообщений и медиа в локальную базу данных.

**Поток:**
1. **UI (SettingsPanel)** – пользователь задаёт параметры: диапазон дат, типы медиа, фильтр по участникам, режим разбивки, экспорт комментариев.  
   Формируется объект `ParseParams`.
2. При нажатии «Начать парсинг» MainWindow запускает `ParseWorker`.
3. **ParseWorker** вызывает `ParserService.collect_data()` с преобразованным `CollectParams`.
4. **Модуль `parser` (`features/parser/api.py`)**:
   - Получает сущность чата, нормализует ID.
   - Итерирует сообщения через `iter_messages`, применяет фильтры по дате, авторам.
   - Для каждого сообщения скачивает медиа (через `_download_media`).
   - Сохраняет сообщения и медиа в БД (таблица `messages`, `media` на диске).
   - Если включены комментарии – отдельно собирает их из linked группы.
5. Возвращает `CollectResult` (количество сообщений, медиа, ошибки).
6. **UI** отображает прогресс и сообщения в логе.

**Задействованные модули:** `SettingsPanel`, `ParseWorker`, `parser`, `DBManager`.

**Передаваемые данные:** `ParseParams` → `CollectParams` → `CollectResult`.

**Известные проблемы / баги (!!!):**
- **Большие видео (>50 МБ) не скачиваются.**  
  _Причина: единый `asyncio.wait_for` не успевает, файл удаляется. Требуется TeleGet._
- **Видео в составе альбомов могут иметь `file_size = 0`, тогда порог не срабатывает.**
- **TeleGet интеграция запланирована, но не реализована** (задача P-задача).

**Статус:** ⚠️ основной функционал парсинга работает, но крупные видео не скачиваются.

---

### 4. Экспорт в целевые форматы (DOCX, MD, JSON, HTML)

**Цель:** Пользователь получает файлы с сообщениями, готовые для просмотра или загрузки в AI-сервисы (NotebookLM).

**Поток:**
1. **UI (SettingsPanel)** – пользователь выбирает форматы экспорта (`docx`, `json`, `md`, `html`), включает опцию AI‑сплит и размер чанка.
2. После парсинга (или отдельно) MainWindow запускает `ExportWorker`, передавая `ExportParams`.
3. **ExportWorker** открывает `DBManager`, для каждого выбранного формата вызывает соответствующий генератор (`DocxGenerator`, `JsonGenerator`, …).
4. **Генератор**:
   - Читает сообщения из таблицы `messages` через `DBManager.get_messages()`.
   - Применяет фильтр по `date_from`/`date_to`, по `topic_id`, по `user_id`.
   - Формирует документ (DOCX через python‑docx, MD/JSON/HTML через шаблоны).
   - Сохраняет файл в `output/<чат>/`.
5. Возвращает список путей к созданным файлам.

**Передаваемые данные:** `ExportParams` → `List[filepath]`.

**Критический баг (!!!):**
- **Экспорт игнорирует выбранный период дат.**  
  _Причина: `ExportParams` не содержит полей `date_from`/`date_to`, а `DBManager.get_messages()` не добавляет условия `WHERE date >= ? AND date <= ?`. В результате экспортируются ВСЕ сообщения из БД, независимо от выбора в UI._  
  **Исправление требует изменений в 3 файлах:** `ui/main_window.py` (передать даты), `features/export/ui.py` (добавить поля), `core/database.py` (обновить SQL‑запрос).

**Другие недочёты:**
- В режиме `split_mode="post"` фильтр по дате не применяется к комментариям (нужно доработать).

**Статус:** 🔴 экспорт по дате сломан. Остальное работает.

---

### 5. Распознавание речи (STT) – голосовые сообщения

**Цель:** Автоматически распознавать голосовые сообщения и сохранять транскрипцию в БД для последующего экспорта.

**Поток:**
1. После парсинга `MainWindow` проверяет, есть ли голосовые сообщения в БД (`get_stt_candidates`).
2. Запускает `STTWorker`.
3. `STTWorker` инициализирует `WhisperManager`, загружает модель `faster‑whisper`.
4. Для каждого голосового сообщения вызывает распознавание, результат сохраняет в таблицу `transcriptions`.
5. По окончании возвращает количество обработанных сообщений.

**Задействованные модули:** `STTWorker`, `WhisperManager`, `faster‑whisper` (внешняя библиотека).

**Известные проблемы / баги:**
- **STT не работает в .exe‑сборке.**  
  _Возможная причина: `faster-whisper` и `numpy` требуют C‑расширения, которые PyInstaller не упаковывает автоматически._  
  - Симптом А: `faster-whisper` установлен в системе, но exe не видит импорт (из‑за `_MEIPASS`).  
  - Симптом Б: при попытке встроить – падает `ImportError` на `numpy` C‑extension.  
  **Рекомендованное решение:** не встраивать STT в exe, а при первом запуске устанавливать `faster-whisper` в отдельное venv (например, в `%APPDATA%\RozittaParser\venv`) и вызывать через subprocess.

**Статус:** 🔴 STT не работает в собранных бинарниках.

## 2. План действий на основе сценариев

| Приоритет | Сценарий | Проблема | Решение |
|-----------|----------|----------|---------|
| 1 | Экспорт (4) | Не фильтрует по датам | Добавить `date_from/date_to` в `ExportParams`, передать в `DBManager.get_messages()`, добавить `WHERE date` |
| 2 | Парсинг (3) | Большие видео не скачиваются | Реализовать адаптивный таймаут / TeleGet (правило >50МБ без wait_for) |
| 3 | STT (5) | Не работает в .exe | Вынести STT в отдельный venv (+ subprocess) или bundling через spec (сложнее) |
| 4 | Авторизация (1) | Импорт tdata с прокси не работает | В `import_from_tdata` передавать настройки прокси из `AppConfig` |
| 5 | Выбор чата (2) | Топ участников не собирает комментаторов | Для каналов собирать авторов из linked группы комментариев |

## 3. Структура данных (БД) – привязка к сценариям

| Таблица | Используется в сценариях | Ключевые поля |
|---------|--------------------------|---------------|
| `messages` | Парсинг (3), Экспорт (4) | `chat_id, message_id, topic_id, date, text, media_path, file_type, reply_to_msg_id, is_comment, post_id, merge_group_id` |
| `chats` | Выбор чата (2), Парсинг (3) | `chat_id, title, type, linked_chat_id` |
| `topics` | Выбор чата (2), Парсинг (3) | `topic_id, chat_id, title` |
| `transcriptions` | STT (5) | `message_id, peer_id, text, model_type` |
| `cached_dialogs` | Выбор чата (2) | `chat_id, title, type, updated_at` |

> Примечание: все даты хранятся в формате `YYYY-MM-DD HH:MM:SS` (строка).

## 4. Контракты модулей (вход/выход)

### `ParserService.collect_data()`
- **Вход:** `CollectParams` (chat_id, topic_id, date_from, date_to, media_filter, download_comments, output_dir, re_download)
- **Выход:** `CollectResult` (success, chat_id, messages_count, comments_count, media_count, errors, db_path)

### `DocxGenerator.generate()` (и аналоги)
- **Вход:** `ExportParams` (chat_id, split_mode, topic_id, user_id, include_comments, period_label, export_formats, ai_split, ai_split_chunk_words, **date_from, date_to**)
- **Выход:** `List[str]` – пути к созданным файлам
